# Workspace Operating Guide

This repository is a consulting project delivery workspace.

## Primary objectives

- analyze project source documents
- design solution architecture
- generate project and implementation documentation
- support customer engagement documentation
- maintain a publishable knowledge base

## Working rules

1. Always read `PROJECT_CONTEXT.md` first.
2. Use existing documents before generating new content.
3. Route outputs into the correct folders.
4. Create draft versions unless explicitly told to finalize.
5. Use professional consulting tone.
6. Keep markdown structure clean and consistent.
7. Prefer concise, reusable documentation.
8. When creating GitHub Pages content, follow the `06-knowledge-portal/docs` structure.

## Routing rules

- RFP analysis outputs go to `02-analysis/rfp-analysis`
- customer and environment analysis go to `02-analysis/customer-analysis`
- delivery estimation and workstream outputs go to `02-analysis/delivery-analysis`
- solution and architecture outputs go to `03-solution`
- delivery planning outputs go to `04-delivery/project-management` or `04-delivery/implementation`
- runbooks go to `04-delivery/runbooks`
- client-ready proposal drafts go to `05-deliverables/proposals/draft`
- design document drafts go to `05-deliverables/design-documents/draft`
- published knowledge pages go to `06-knowledge-portal/docs`

## Writing expectations

- executive summary should be short
- write for both decision makers and delivery teams
- use headings that are clear and searchable
- use tables only when they improve clarity
- preserve traceability to source requirements where practical
